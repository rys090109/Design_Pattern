# 방과후 A — 디자인 패턴 정리

---

## 1. 디자인 패턴이란?

| 개념 | 설명 |
|------|------|
| **디자인 패턴** | 반복되는 설계 문제에 대한 23가지 검증된 해결책 |
| **설계(디자인)** | 그룹과 관계를 정의하는 것 |
| **아키텍처(구조)** | 시스템 전체의 구조적 설계 |

### 그룹의 구성 요소

```
클래스 + 함수 + 메소드 = 그룹
```

- **클래스** : 데이터와 동작을 묶는 단위
- **함수** : 명령어 묶기
- **메소드** : 클래스에 속한 함수

---

## 2. 생성자의 불편함

- **객체 강제 생성** — `new` 키워드로 매번 새 객체를 만들어야 함
- **초기화 불편함** — 생성 시점에 모든 정보를 알아야 함

→ 이 문제를 해결하기 위한 패턴들이 등장

## 4. 패턴 그룹별 분류

### 4-1. 미리 생성 (1, 2, 3)
> "언제 재사용하고 언제 생성할 것인가?"

| 패턴 | 전략 |
|------|------|
| Singleton | 1개만 생성, 계속 재사용 |
| Flyweight | N개 미리 생성, 캐시처럼 재사용 |
| Prototype | 기존 객체를 복제해서 사용 |

### 4-2. 분리 (4, 5, 6, 7)
> 분리 기준:
> 1. **관심사** — SRP (단일 책임 원칙)
> 2. **가변 vs 불변** — OCP (개방-폐쇄 원칙)

| 패턴 | 분리 방식 |
|------|-----------|
| Builder | 생성 로직 분리 |
| Template Method | 불변 골격은 상속, 가변 부분만 오버라이드 |
| Strategy | 가변 알고리즘을 주입 |
| Bridge | 상속 + 주입 결합 |

### 4-3. 동일시 (9, 10, 11)
> 서로 다른 것을 같은 타입으로 취급

| 패턴 | 동일시 대상 |
|------|-------------|
| Composite | 요소(파일) ↔ 집합(폴더) |
| Command | 명령 ↔ 객체 |
| Decorator | 장식 ↔ 내용물 |

### 4-4. 상태 (8, 14, 17)

| 패턴 | 역할 |
|------|------|
| State | 현재 상태에 따라 행동 |
| Observer | 상태 변화를 관찰/통보 |
| Memento | 상태를 기억하고 복원 |

### 4-5. 연결 (15, 20, 19)

| 패턴 | 역할 |
|------|------|
| Iterator | 컬렉션을 순차적으로 연결 |
| Chain of Responsibility | 처리자들을 체인으로 연결 |
| Facade | 서브시스템을 단순 인터페이스로 연결 |

## 5. SOLID 원칙 (OOP)

| 원칙 | 설명 | 관련 패턴 |
|------|------|-----------|
| **S** — SRP (단일 책임 원칙) | 클래스는 하나의 이유로만 변경 | Strategy, Bridge |
| **O** — OCP (개방-폐쇄 원칙) | 확장에 열려 있고, 수정에 닫혀 있음 | Template Method |

| **D** — DIP (의존 역전 원칙) | 추상에 의존, 구체에 의존 X | — |

## 6. 주요 패턴 상세

### Singleton (싱글톤)
> 미리 만들고, 1개만 재사용

```java
class Singleton {
    static Singleton s = new Singleton();  // 미리 생성
    private Singleton() {}                 // 외부 생성 차단

    static Singleton getInstance() {
        if (s == null)
            return new Singleton();
        return s;                          // 기존 인스턴스 반환
    }
}
```

### Template Method (템플릿 메서드)
> 공통 골격(템플릿)을 **상속**받아 완성

- 불변 부분 → 부모 클래스에 고정
- 가변 부분 → 자식 클래스에서 오버라이드

### Strategy (전략)
> 알고리즘을 **주입**받아 완성

- 실행 중 알고리즘 교체 가능
- 상속 대신 합성(Composition) 사용

### Decorator (데코레이터)
> **장식과 내용물을 동일시** (같은 인터페이스)

- SOLID의 **L (LSP)** 와 연관
- 기능을 동적으로 추가/조합 가능
- 예: `Coffee` → `MilkDecorator(Coffee)` → `SugarDecorator(MilkDecorator(Coffee))`

### Composite (컴포지트)
> **요소(단일)와 집합(복합)을 동일시**

- 파일 시스템: 파일(요소) = 폴더(집합) → 둘 다 같은 인터페이스
- `엔티티 = 요소 && 집합`

## 7. 타입 시스템 (Composite 이해를 위한 기초)

> "모든 값에는 타입이 있고, 타입을 모르면 값을 모른다."

```
1kg  →  1 (값) + kg (타입)
```

| 구분 | 특징 | 예시 |
|------|------|------|
| **기본형** | 1개, 요소, 파일 | `int`, `char`, `boolean` |
| **참조형** | 기본형 * n, 그룹, 폴더 | `Class`, `Array`, `String` |

```java
class Time {
    int hour;
    int min;
    int sec;
}

String str = null;  // null의 타입은 null
```

> **컴파일러** = if문 덩어리  
> 컴파일 → 번역 → 문법 체크(if문 사용)
